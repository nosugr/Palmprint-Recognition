# 软硬件连接说明 · 掌纹识别门禁系统

> 面向：软件组（你）+ 嵌入式组（写固件的同学）。
> 目的：讲清楚 PC 上的算法/前后端，怎么和 STM32 + 电磁锁/LED/蜂鸣器连起来。
> 配套契约见 [`INTERFACE_CONTRACT.md`](./INTERFACE_CONTRACT.md) 契约 A。

---

## 0. 一句话结论

PC 软件和硬件之间**只靠一根 USB 线 + 一套文本命令**连接。
双方各自守住"命令约定"就能连上，**谁都不用看对方的源码，代码也不用放在同一台电脑上。**

---

## 1. 常见误解先澄清（重要）

**❓ 算法代码和固件代码，要放在同一台电脑上吗？**
**不用。而且本来就不可能在一起。**

固件最终不是"跑在电脑上"的，它跑在 STM32 芯片里。理清三个角色：

| 角色 | 作用 | 运行什么 |
|------|------|---------|
| **嵌入式组的电脑** | 只用来**写固件 → 编译 → 烧录进芯片**。烧完就退场，跟日常运行无关 | Keil / STM32CubeIDE |
| **STM32 芯片（开发板）** | 烧录后固件**跑在芯片里**，接收串口命令、驱动锁/灯/蜂鸣器 | 固件（已烧录） |
| **你的电脑** | 跑算法 + Flask 后端 + Vue 前端，通过 USB 线给芯片发命令 | 本仓库的 Python/前端 |

```
嵌入式组电脑 ──(USB线·一次性烧录)──► STM32芯片 ──(USB线·长期插着)──► 你的电脑
   写固件、编译                        固件在这里跑              跑算法/前后端
   烧完就拔走                                                   日常就用这台
```

**❓ 要把我的前后端项目发给嵌入式组吗？**
**不用。** 他不需要你的 Python 代码。你俩唯一要共享的是**串口命令的约定**——把本文件（或 `INTERFACE_CONTRACT.md`）发给他即可。

---

## 2. 职责切分

| 范围 | 负责方 |
|------|--------|
| 视觉算法、Flask 后端、SQLite、Web 前端、串口命令的**发送侧** | 软件组（本仓库，你的电脑） |
| STM32 固件、电磁锁/LED/蜂鸣器电路、串口命令的**接收与执行侧** | 嵌入式组 |

边界就是那根 USB 串口线。线以上是你，线以下是他。

## 3. 物理连接（接线图）

### 3.1 总览

```
                         ┌──────────────────────────────┐
                         │        你的电脑 (PC)          │
                         │  摄像头→算法→判决→Flask后端   │
                         └───────────────┬──────────────┘
                                         │ USB 线 (Type-C)
                                         │ ① 给开发板供电(5V)
                                         │ ② 串口通信(USB-CDC, 115200)
                         ┌───────────────▼──────────────┐
                         │      STM32 开发板 (F407)      │
                         │  固件解析命令，控制下面4路    │
                         └──┬────────┬────────┬─────────┘
                  GPIO控制  │        │        │  GPIO控制
            ┌───────────────┘        │        └───────────────┐
            │                        │                        │
      ┌─────▼─────┐          ┌───────▼───────┐         ┌──────▼──────┐
      │  继电器模块 │          │  蜂鸣器模块    │         │ 绿LED / 红LED│
      │ (控制电磁锁)│          │ (单响/双响)    │         │ (+限流电阻)  │
      └─────┬─────┘          └───────────────┘         └─────────────┘
            │ 继电器开关 12V 回路
   ┌────────▼─────────┐        ┌─────────────────────┐
   │   电磁锁 (12V)    │◄───────│  可调电源 (调到12V)  │
   │                  │        │  独立给锁供电        │
   └──────────────────┘        └─────────────────────┘
```

### 3.2 为什么电磁锁要单独用可调电源，不能从开发板取电？

电磁锁工作电流大（通常 0.3~1A、12V），**STM32 开发板的引脚带不动，硬接会烧板子**。
正确做法：电磁锁由**可调电源**单独供 12V，STM32 只通过**继电器**去"通断"这条 12V 回路——
STM32 出的是一个小小的控制信号（几毫安），真正的大电流走电源→继电器→锁，不经过芯片。

### 3.3 接线对照表

> ⚠️ 下表的 STM32 引脚号是**示例/约定**，最终由**嵌入式组在固件里定义**，以他们的为准。
> 你不用关心具体引脚——你只管发命令，引脚是他那边的事。

| 信号 | STM32 引脚(示例) | 接到 | 说明 |
|------|------------------|------|------|
| 串口 TX/RX | USB 口 (USB-CDC) | 你的电脑 USB | 用板载 USB 即可，免接杜邦线 |
| 继电器控制 | PA0 → 继电器 IN | 继电器模块 | 高/低电平触发，看模块 |
| 蜂鸣器 | PA1 → 蜂鸣器 IN | 蜂鸣器模块 | 单响=成功，双响=失败 |
| 绿 LED | PA2 →(限流电阻)→ LED+ | 绿灯 | 验证成功亮 |
| 红 LED | PA3 →(限流电阻)→ LED+ | 红灯 | 验证失败亮 |
| **共地** | GND | 继电器/蜂鸣器/LED/可调电源 **GND 全部接到一起** | **关键！不共地会乱跳或不动作** |

**继电器与电磁锁那一侧（强电回路）：**

```
可调电源(+12V) ──► 继电器 COM
继电器 NO ──► 电磁锁(+)
电磁锁(-) ──► 可调电源(GND)
```

（用常开 NO：平时断电锁着，收到 UNLOCK 时继电器吸合通电开锁。具体看你用的是断电锁还是通电锁，逻辑可能反过来，由嵌入式组配合固件确定。）

## 4. 它们之间"说什么话"（串口协议）

物理层：USB-CDC 虚拟串口，**115200** 波特率，ASCII 行协议，每条命令以 `\n` 结尾。

**PC → STM32（你发，他执行）**

| 命令 | 含义 | STM32 动作 |
|------|------|-----------|
| `UNLOCK <ms>\n` | 开锁 ms 毫秒 | 继电器吸合，到时自动回锁 |
| `OK\n` | 验证成功 | 绿灯亮 + 蜂鸣器单响 |
| `FAIL\n` | 验证失败 | 红灯亮 + 蜂鸣器双响 |
| `PING\n` | 健康检查 | 必须回 `PONG\n` |

**STM32 → PC（他回，可选但推荐）**

| 响应 | 含义 |
|------|------|
| `PONG\n` | 对 `PING` 的应答，用于后端 `/api/health` 判断硬件在不在线 |
| `ACK <CMD>\n` | 已收到并执行 |
| `ERR <msg>\n` | 执行出错 |

> 这些命令由本仓库 `hardware/bridge.py` 的 `SerialBridge` 发出，
> 在 `server/routes.py` 的 `/api/verify` 命中后调用 `unlock()` + `indicate(True)`。

## 5. 联调步骤（从零到能开锁）

1. **嵌入式组**：把实现了上面协议的固件烧进 STM32（在他自己电脑上完成，用 Keil/CubeProgrammer）。
2. **接硬件**：按第 3 节接线图把继电器、电磁锁、可调电源(调到12V)、蜂鸣器、LED 接好，**所有 GND 共地**。
3. **插线**：STM32 用 USB 线插到**你的电脑**。
4. **认端口**：设备管理器里看新出现的 COM 口（如 `COM3`）。
   - 在 `config.py` 把 `SERIAL_PORT = "auto"` 改成具体的 `"COM3"` 更稳（多个串口时 auto 可能抓错）。
5. **切到真硬件模式**：现在默认走 `MockBridge`（只打日志、不真开锁）。
   要接真硬件，需让 `server/app.py` 里 `create_bridge(use_serial=True)` 生效。
6. **启动后端**，访问 `/api/health` 看 `"hardware": true` 是否点亮（这一步靠 PING/PONG）。
7. **走一次 verify**：命中后应听到蜂鸣器单响、绿灯亮、锁"咔"地开。

## 6. 常见坑（接上了却不动，先查这几条）

| 现象 | 可能原因 |
|------|---------|
| `/api/health` 里 hardware 一直是 false | 固件没回 `PONG`；或波特率不是 115200；或 COM 口选错 |
| 完全没反应 | GND 没共地；USB 线是"只充电不传数据"的劣质线 |
| 命令收不到/乱码 | 行尾固件按 `\r\n` 解析，而 PC 发的是 `\n`；波特率不一致 |
| 灯响了但锁不开 | 锁没单独供电（别想从开发板取电）；继电器常开/常闭接反；断电锁vs通电锁逻辑反了 |
| 开锁不回锁 | `UNLOCK` 的时长固件没用 PC 传的值，写死了 |

---

## 7. 给嵌入式组的「4 点确认清单」

接线前后，和嵌入式组把这 4 件事敲定（契约里标的 TODO）：

1. **波特率**用不用 115200？
2. **行尾**用 `\n` 还是 `\r\n`？
3. 要不要实现 `PING→PONG`（不实现的话后端健康检查永远显示硬件离线）？
4. **开锁时长**由 PC 传（`UNLOCK <ms>`）还是 STM32 固定？

这 4 点对齐了，软硬件基本就能一次连通。
